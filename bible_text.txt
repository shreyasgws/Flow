6: PART ONE — IDENTITY
7: 1. THE IDEA
8: Flow is a calm operating system for daily momentum.
9: Not a productivity dashboard. Not a hustle tool. Not a habit tracker. Not a gamified planner.
10: A place to plan your day without turning your life into a productivity system.
11: No health tracking. No mood logging. No gamification. No streaks. No guilt. No pressure. No scores.
12: Just a clean, honest, breathing space to plan your day, move through it, and quietly reflect on how it went.
13: The Core Belief
14: A productivity app should feel like a scratchpad with structure — not a contract you sign every morning.
15: What Makes Flow Different
16: Tasks are completely free-form — type and go, nothing is ever forced
17: Categories exist but are never demanded
18: Plans are suggestions, not commitments
19: The interface feels alive, not clinical
20: The app has soft memory but never locks you in
21: Tasks are suggestions, not obligations
22: The app gets out of your way
23: Drift captures thoughts before they become tasks
24: Review is reflection, not judgment
25: The timeline IS the identity of the app
26: What Flow Should NEVER Feel Like
27: Corporate or dashboard-heavy
28: Overly analytical or gamified
29: Stressful, rushed, or hyper-optimized
30: Cluttered, robotic, or cold
31: Like a system you must maintain
32: What Flow Should Always Feel Like
33: Ambient and breathable
34: Emotionally safe and warm
35: Frictionless and tactile
36: Alive and gently aware
37: Non-judgmental and forgiving
38: Like opening a notebook at dawn
39: 2. THE PHILOSOPHY
40: Freedom First
41: Every task starts as a line of text. Type it, hit enter, it is there. No category required. No icon required. No Flow Section required. Add those later, or never — the task looks complete either way. Nothing feels incomplete without metadata.
42: Soft Memory, Not a Library
43: Flow remembers what you have typed before and surfaces gentle ghost suggestions when you start typing. Suggestions fade after 7 days of non-use. Nothing piles up. Nothing locks you in. There is no task library you must maintain.
44: Carry Forward, Your Choice
45: Unfinished tasks from yesterday appear as a soft morning nudge. One tap to carry forward. One tap to dismiss. Never automatic. Never guilt-inducing.
46: Momentum Over Completion
47: Flow is not about finishing everything. It is about staying in motion. A day where you completed 4 of 7 things is not a failed day. It is a day where 4 things moved forward.
48: Review Without Pressure
49: End of day is a glance — what got done, what did not. An emotional tone, not a score. A reflection, not a report.
50: Continuity, Not Addiction
51: The app builds continuity. Not streaks. Not chains. Not guilt loops. Just a gentle daily rhythm that makes planning feel like a natural part of the day.
52: Momentum-Centric, Not Task-Centric
53: The most important architectural decision in Flow.
54: Task-centric apps ask: "Did you finish your tasks?"
55: Flow asks: "Are you still moving?"
56: The home screen is not a task list. It is a live view of where your day stands.
57: The timeline is not a schedule. It is a record of momentum in progress.
58: The review is not a report. It is a quiet reflection on how the day moved.
59: The weekly view is not analytics. It is a memory of how the week felt.
60: Every design decision in Flow flows from this distinction.
61: 3. WHAT THIS APP IS NOT
62: Not a habit tracker (no streaks, no chains, no daily goals)
63: Not a health tracker (no steps, water, calories, mood)
64: Not a project manager (no subtasks, no assignees, no Kanban)
65: Not a calendar (no event invites, no time-blocking grid)
66: Not a journaling app (carry reason is a quick note, not a journal)
67: Not a social app (no sharing, no comparison, no public profiles)
68: Not a hustle tool (no productivity scores, no leaderboards, no analytics)
69: Not a note-taking app (Drift is not a notes app)
70: Flow is a personal daily momentum companion. Nothing more. Nothing less.

80: 1. FRONTEND FRAMEWORK
81: Next.js — App Router
82: Flow needs smooth page transitions, installability, mobile and desktop responsiveness, high-performance rendering, and scalable architecture. Next.js App Router provides layouts, transitions, nested systems, ambient environment layers, edge optimization, and PWA support.
83: Recommended folder structure:
84: app/ components/ features/ stores/ hooks/ lib/ styles/ motions/ tokens/ types/ utils/
85: A modular structure prevents chaos as Flow grows across Drift, reflections, Flow Sections, ambient environments, focus mode, undo systems, and sync systems.
86: ---
87: 2. UI LIBRARY
88: React
89: Flow's UI changes constantly: ambient transitions, task movement, gesture states, drag systems, atmosphere adaptation, and focus transitions. React is ideal for reactive emotional interfaces.
90: Avoid massive component files, giant prop chains, and overuse of context. Prefer modular components, Zustand stores, and feature-based organization.
91: ---
92: 3. STYLING SYSTEM
93: Tailwind CSS + Semantic Tokens
94: Flow depends on atmosphere systems, adaptive spacing, motion states, dynamic themes, and ambient layering. Tailwind makes token-driven design systems much easier.
95: CRITICAL: Use semantic tokens, never hardcoded values.
96: surface-primary | surface-secondary | surface-elevated
97: text-primary | text-secondary | text-muted
98: ambient-glow | ambient-noise | ambient-tint
99: spacing-density | motion-intensity | blur-level
100: Theme architecture: light theme + dark theme + ambient modifiers + environment modifiers + motion modifiers.
101: ---
102: 4. ANIMATION SYSTEM
103: Motion (formerly Framer Motion)
104: Flow depends on emotional pacing, tactile interactions, atmosphere, calm transitions, and continuity feeling. Motion is one of the best libraries for emotionally responsive UI.
105: Handles: layout transitions (tasks moving between Flow Sections), shared element transitions (task opening into focus mode), ambient motion (subtle atmospheric drifting), and gesture systems (drag-to-convert Drift entries).
106: Motion timing guide:
107: 120ms micro interactions | 220ms UI transitions | 400ms environmental changes | 800ms+ ambient atmosphere
108: Avoid cinematic motion, flashy effects, and constant movement. Prefer opacity, scale, soft transforms, easing, and spacing shifts.
109: ---
110: 5. STATE MANAGEMENT
111: Zustand
112: Redux is too heavy for Flow. Flow needs lightweight interaction state, fast updates, minimal boilerplate, and flexible stores.
113: Recommended stores: taskStore, driftStore, environmentStore, focusStore, syncStore, undoStore, settingsStore, reflectionStore.
114: The environment store becomes the heart of Ambient Mode:
115: environmentStore = { theme, ambientMode, motionIntensity, atmosphereLevel, spacingDensity, contrastLevel }
116: Use React Query (TanStack Query) alongside Zustand for server-state synchronization with Supabase.
117: ---
118: 6. LOCAL-FIRST DATABASE
119: IndexedDB + Dexie.js
120: THIS IS ONE OF THE MOST IMPORTANT DECISIONS.
121: Flow should be local-first: interactions happen instantly with no loading feeling, sync happens quietly later. Without local-first, typing lag and drag delay destroy Flow's emotional philosophy of immediate responsiveness.
122: Dexie.js provides simpler APIs, querying, transactions, and versioning over raw IndexedDB.
123: Local-first sync architecture:
124: User Action → Local DB Update → Instant UI Update → Background Sync Queue → Supabase Sync
125: ---
126: 7. BACKEND
127: Supabase
128: Handles cloud sync, auth, backups, persistence, and cross-device continuity. Supabase provides PostgreSQL, auth, storage, realtime subscriptions, and row-level security.
129: Flow has relational data, historical continuity, recurring systems, reflections, undo stacks, and export systems — all deeply relational. PostgreSQL handles this beautifully.
130: ---
131: 8. AUTHENTICATION
132: After successful migration: delete all anonymous-ID records from Supabase. Show a single message: 'Your data is saved to your account.'
133: When the user chooses to sign in with Google: call supabase.auth.linkIdentity({ provider: 'google' }). If link succeeds, Supabase merges the accounts and the user keeps their data. If link fails, fall back to manual migration.
134: Flow uses anonymous-first onboarding by default. On first open, a Supabase anonymous session is created immediately (supabase.auth.signInAnonymously()). The user experiences the full product without any sign-in wall. Data saves to both IndexedDB (under the anonymous user ID) and Supabase.
135: Architectural Decision: Anonymous-first onboarding
136: Supabase Auth
137: Recommended methods: Email, Magic Link, Google OAuth.
138: CRITICAL: Use anonymous-first onboarding. Let users experience Flow before signup. This strongly fits the philosophy.

291: 7. NAVIGATION PHILOSOPHY
292: The biggest goal of Flow's navigation: users should never feel "inside menus."
293: Instead, the app should feel like moving through layers of your day.
294: Home is the center of gravity. Everything else emerges from it, supports it, and returns to it.
295: Flow should never feel like:
296: A dashboard app
297: A productivity control panel
298: A feature-heavy workspace
299: It should feel like:
300: Open app → understand your day instantly → move naturally through tasks → focus deeply when needed → reflect lightly later → return tomorrow without pressure
301: 8. BOTTOM NAVIGATION — 4 TABS ONLY
302: [ Home ]  [ Week ]  [ Focus ]  [ Settings ]
303: That is it. No more, no less.
304: Why 4 tabs and not 5:
305: Review is not a primary destination. It is discovered by tapping a past date.
306: Drift is not a primary destination. It lives as the floating Drift button (always present).
307: Templates and Categories live inside Settings.
308: Adding a fifth tab would fragment attention away from Home.
309: Tab: Home
310: The living workspace. Where users spend 85–90% of their time.
311: Purpose: current day, current momentum, task flow, planning, continuity.
312: Tab: Week
313: Reflection and navigation.
314: Purpose: emotional overview of the week, navigating between dates.
315: NOT an analytics dashboard.
316: Tab: Focus
317: Dedicated immersive state. Enters focus mode when a task is active.
318: When no task is in focus: tapping Focus tab shows the last focused task with a "Resume?" prompt, or prompts the user to pick a task from today's plan.
319: This tab deserves primary navigation because it is a deliberate mental mode shift — not just a feature.
320: Focus tab behavior:
321: Active session: tab glows softly with accent color, shows task title truncated
322: No active session: tab is dimmed, tap shows picker
323: During active focus: tapping the tab returns to the full focus screen
324: Tab: Settings
325: Customization layer. Everything secondary lives here.
326: Profile, theme, day start time, Flow Section names, categories, templates, notifications.

362: 10. DRIFT BUTTON — CUSTOM DANDELION ICON
363: The Drift button has a completely custom icon: an animated dandelion.
364: No other app uses this icon. It is exclusively Flow's.
365: Icon Design
366: 8 stems radiating from a center core
367: Each stem has a circular seed tip at its end
368: Stems going upward (top, top-left, top-right) are brightest and most opaque
369: Stems going downward (bottom, bottom-left, bottom-right) are most faded
370: This gradient of opacity represents seeds already drifting away
371: The asymmetry makes it feel organic, not geometric
372: Specific Stem Opacities (8 stems, clockwise from top)
373: Top: 90% opacity, length 11px
374: Top-right: 76% opacity, length 10px
375: Right: 60% opacity, length 10px
376: Bottom-right: 44% opacity, length 9px
377: Bottom: 30% opacity, length 8px
378: Bottom-left: 26% opacity, length 9px
379: Left: 50% opacity, length 10px
380: Top-left: 66% opacity, length 10px
381: Animation: Idle Breathing (always running)
382: The dandelion breathes continuously while visible.
383: Each of the 8 stems extends and contracts independently
384: Each stem has a unique phase offset (evenly distributed across 2π)
385: This means stems never all move in sync — they feel organic
386: Stem extension: ±1.3px from base length per cycle
387: Tip size: ±0.35px per cycle
388: Center core: ±0.28px per cycle
389: Speed: t += 0.022 per frame (extremely calm)
390: All animations use Math.sin with staggered phase offsets
391: The user feels the breathing more than sees it. It is alive without being distracting.
392: Animation: Tap Burst
393: When the user taps the Drift button:
394: Phase 1 — Scatter out (0 to 380ms):
395: All 8 stems extend outward rapidly (easeOut cubic)
396: Maximum extra length: 9px beyond base length
397: Each stem has a slight stagger offset (i * 30ms) for organic scatter
398: Tip opacity decreases slightly to 70%
399: Phase 2 — Hold at max scatter (380ms to 500ms):
400: Stems held at maximum extension
401: Tips at 70% opacity
402: Phase 3 — Spring back (500ms to 1050ms):
403: Stems return to base length with spring overshoot
404: Spring function creates a satisfying elastic bounce-back
405: Tips return to base opacity
406: Center core pulses slightly larger during return
407: Total burst duration: ~1050ms
408: The sheet opens 180ms after the tap (slight delay so burst begins first).

1030: Each day has a visual weight the UI naturally expresses — never labeled, always felt.

1034: Total task count

1040: Timeline line: thinner (1px), more delicate
1041: Background feels more spacious

1043: Balanced day (3–6h estimated, mixed friction):
1044: Task card margin-bottom: 8px
1045: Flow Section header padding-top: 24px
1046: Default spacing, no special treatment
1047: Heavy day (over 6h estimated OR 7+ tasks OR 3+ heavy friction tasks):

1312: 34. MOTION AND ANIMATION
1313: Every animation serves meaning. Nothing animates for decoration alone.
1314: Core Principle
1315: Every animation serves meaning. Nothing animates for decoration alone. Motion in Flow communicates state change, provides spatial orientation, and reinforces the calm emotional identity. All animations are optional via Reduced Motion (Section 44, Feature 23).
1316: Animation Architecture
1317: All animations use CSS transforms (translate, scale, opacity) on GPU-composited layers. Never animate layout properties (width, height, top, left, margin, padding). Use will-change: transform on animated elements. Prefer CSS animations over JavaScript-driven animation for ambient/looping motion.
1318: Timing and Easing
1319: Default easing: cubic-bezier(0.22, 1, 0.36, 1) — a custom ease-out curve for almost all UI motion.
1320: Spring easing (for tactile interactions): stiffness 120, damping 14, mass 1.
1321: Duration scale: 100ms (micro-interactions, button press), 200ms (hover, ambient), 300ms (card expand, sheet open), 400ms (page transitions, focus entry), 600ms (background crossfade, theme switch), 1000ms+ (ambient breathing, particle cycles).
1322: Ambient Motion
1323: Background layers: 20–40s cycle, CSS keyframes on transform, ±2% scale oscillation.
1324: Particles: 2–3 ultra-fine glow dots, 40s drift cycle, randomized phase. CSS animation with very long duration.
1325: Drift dandelion: continuous breathing animation, 8 stems with staggered Math.sin phase, speed t += 0.022/frame.
1326: All ambient motion pauses when Reduced Motion is enabled. Ambient motion also pauses during Focus Mode to avoid distraction.
1327: Page Transition Animations
1328: Home ↔ Week: slide horizontal (300ms), content crossfade.
1329: Home ↔ Focus: scale + fade (400ms), chrome fade-out (300ms).
1330: Home ↔ Review: zoom into date (300ms spring), background crossfade.
1331: Home ↔ Settings: slide vertical (250ms).
1332: All transitions share a coordinated timing contract — chrome, content, and background animate as a unified system.
1333: Stagger Animations
1334: Weekly view day cards: 40ms stagger delay left-to-right, fade + translateY.
1335: Task list on load: 30ms stagger per card, fade in.
1336: Composer metadata icons: 30ms stagger, fade up.
1337: Bottom sheet content: 50ms stagger per row.
1338: Action menu buttons: 30ms stagger, slide from right.
1339: Respecting System Preferences
1340: Read prefers-reduced-motion media query. When reduced motion is preferred: disable all ambient/looping animations, replace spring physics with 200ms linear fades, disable stagger delays, disable particle effects, disable background breathing, keep only essential state-change animations (checkmark fill, card expand) at 1.5× duration to avoid jarring jumps.
1341: Animation Performance Budgets
1342: Maximum simultaneous animated elements: 8 (ambient) + 4 (interactive). Maximum blur radius on animated elements: 12px during animation, 20px at rest. Particle count: max 3. No animation on elements below the fold until they enter viewport. All animations must maintain 60fps on mid-range devices (Snapdragon 700 series, iPhone X equivalent). Use requestAnimationFrame for JS-driven animation, never setInterval.

1831: 48. DATA SYNC STRATEGY
1832: Core Rule: Local-First, Offline-Fallback
1833: Revised Architecture Decision: Local-first with Supabase as sync target. IndexedDB is the source of truth. Every write goes to IndexedDB first and immediately. The sync worker replays operations to Supabase in the background. Supabase confirms what IndexedDB already has. Rollbacks never happen because local state is always the committed state. The sync worker is a background process that does not block any user interaction.
1834: Local Write Flow (default, always preferred)
1835: User performs any action
1836: UI updates instantly via optimistic local state (0ms perceived delay)
1837: Supabase write fires immediately in background (within milliseconds)
1838: On Supabase success: local optimistic state confirmed, no UI change
1839: On Supabase error: auto-retry up to 3 times with exponential backoff (1s, 2s, 4s)
1840: If all 3 retries fail: action queued to IndexedDB sync queue, subtle error shown
1841: Offline Fallback Flow
1842: Triggered when: navigator.onLine = false OR Supabase request times out after 5s.
1843: UI updates instantly (same optimistic behavior)
1844: Write queued to IndexedDB sync queue with full action payload + timestamp
1845: Subtle offline indicator appears: 6px dot top-right, --warn color, "Working offline" tooltip
1846: User continues working normally — all interactions still feel instant
1847: When connection restores: sync queue flushes automatically (oldest first)
1848: Each queued action replayed against Supabase
1849: Conflicts (two devices edited same record while offline): last-write-wins by timestamp
1850: Offline indicator disappears when queue is empty and Supabase confirms
1851: Subtle Save & Sync Status System (Additional Feature)

1947: Feature 1: Day Templates
1948: Architecture
1949: Day templates store a snapshot of a day's plan structure — not specific task instances. When a template is applied, new plan_task rows are created in the target day_plan. The original tasks are never linked to the template after application.
1950: Template data structure (stored in template_tasks jsonb):
1951: { title: string, slot: uuid (FK to flow_sections.id) | null, category_id: uuid | null, time_estimate: number | null, friction_level: 'light' | 'medium' | 'heavy' | null }
1952: Template application flow:
1953: 1. User taps "Apply Template" on /plan/:date. 2. Bottom sheet shows list of templates (name + task count). 3. User selects a template. 4. Tasks are appended immediately. An undo snackbar appears: "Added [N] tasks from [template name]" [Undo]. 5. Tasks are appended to the current plan in order (preserving slots). 6. A soft success animation plays: tasks fade in with stagger (30ms each). 7. Day Pulse recalculates. 8. Day Weight recalculates.
1954: After application: all tasks are fully editable independently. No link back to the template exists. Templates are only starting points.
1955: Template source: saved from any day's plan via long-press on empty area or "..." menu in the top-right. The template captures the current state of tasks at the moment of saving. Future edits to the source day do not affect saved templates.
1956: Maximum templates: 30. If the user hits the limit, the save button shows "Template limit reached (30). Delete an existing template first."
1957: Save any day's plan as a named template. Apply to any future date in one tap. Tasks appended (never replace existing). Fully editable after applying. Stores: task titles, slots, categories, time estimates, friction levels, mode. Never locks — applying a template is only a starting point. Managed in Settings → Day Templates.
1958: Feature 2: Drag to Reorder
1959: Architecture
1960: Drag-to-reorder operates on the plan_tasks table's order integer column. On drop, a batch update is sent to Supabase updating the order values for all affected tasks in a single transaction.
1961: Order values are stored as integers (0, 1, 2, …) per plan. Gaps are allowed (e.g., 0, 5, 10) to accommodate inserts. On explicit reorder, the order values are normalized (compacted) to sequential integers.
1962: Cross-section drag: if a task is dragged from Morning to Afternoon, the task's slot field updates to 'afternoon' and its order becomes the last position in the afternoon slot. The section boundary is indicated by a visual gap with a drop-zone highlight line.
1963: Undo support: all reorder actions are added to the undo stack. If the user shakes the device (iOS) or taps Undo in the snackbar, the task returns to its previous position and slot. The reorder animation replays in reverse.
1964: Long press (300ms) → card lifts with spring animation → drag up or down → spring drop. Works in both Routine (timeline) and Checklist modes. Order saves immediately to Supabase (optimistic). Other cards animate out of the way using spring physics during drag.
1965: Feature 3: Time Estimates
1966: Architecture
1967: Time estimates are stored as integers (minutes) in the time_estimate column. They are optional and nullable. The estimate is displayed as a pill in the expanded task view.
1968: Display format: < 60 minutes → "Xmin" (e.g., "25min"), >= 60 minutes → "Xh Ymin" (e.g., "1h 30min"), >= 480 minutes → "Xh" (e.g., "8h").
1969: Flow Section header aggregation: "Morning · ~2h 30min" sums all time_estimate values for tasks in that section. The total is shown in the section header, right-aligned, DM Sans 11px, --text-secondary.
1970: Day total: the Day Pulse may reference the total estimated time for the day. The Day Weight system uses time estimates as one of three inputs (see Section 23).
1971: No validation or warnings: if the user adds 20 tasks each with 2h estimates, the app does not warn about overload. The Day Weight system will express the density visually, but no alert is shown.
1972: Optional per-task in minutes. Shown as a tiny time pill in expanded view. Flow Section headers show total estimated time: "Morning · ~2h 30min". Day Pulse references total estimated time for the day. Used in Day Weight calculation. Never required. Never blocks task creation.
1973: Feature 4: End of Day Review
1974: Architecture
1975: The review page is a read-only representation of a past day's plan. It is generated from the plan_tasks + task_instances for that date, joined with the tasks table for metadata.
1976: Data source: the day_plan for the given date is fetched with all associated plan_tasks and task_instances. If no day_plan exists for that date, the empty review state is shown (see Section 64).
1977: Completion ring calculation: completed_count / total_count. The ring is a stroked SVG circle with stroke-dasharray and stroke-dashoffset animation. Arc angle: 0 to 360 degrees, animated on load (300ms ease-out).
1978: Carry-forward from review: the bottom "Carry unfinished to tomorrow" button opens a sheet listing unfinished tasks. User selects which to carry. Tasks are added to tomorrow's plan (or today's if tomorrow hasn't been initialized yet). If tomorrow's day_plan doesn't exist, it is created on carry-forward.
1979: Review is always read-only: no editing, no reordering, no completion toggling from review. The review is a record, not a workspace.
1980: Accessed at /review/:date. Always read-only. Soft completion ring, emotional summary text, tasks by slot. Carry-forward option for unfinished tasks. Tone is always calm and non-judgmental. Accessed via date strip (tap past date) or weekly view (tap past day card).
1981: Feature 5: Weekly View
1982: Architecture
1983: The weekly view queries day_plans for all dates in the visible week. For each day_plan, it computes: total task count, completed count, emotional label (based on completion ratio and metadata), slot-level completion breakdown.
1984: Lazy loading: only the visible week is fetched on mount. Adjacent weeks are pre-fetched when the user swipes (pre-fetch one week ahead in the swipe direction). Historical weeks (past) remain cached after fetch. Future weeks are not cached (plans may change).
1985: Day card rendering: each card is a fixed-width flex item (1/7 of the container minus gaps). On screens smaller than 375px, the date number font size reduces to 14px and the emotional label is hidden to prevent overflow.
1986: Tap navigation: tapping a day card routes to /home (today), /plan/:date (future), or /review/:date (past). The transition is a zoom animation where the tapped card scales to fill the screen (300ms spring).
1987: Historical scroll: the weekly view supports infinite scroll backward. Each swipe left triggers a fetch for the previous week. There is no query limit — the user can scroll back as far as they have data. If the fetch returns zero results, the week renders with all empty-state cards.
1988: 7-day emotional memory grid. Each card: day name, date, emotional label, completion tint bar. Swipe between weeks. Tap any day to plan or review. No harsh percentages. No graphs. No scores. Stagger animation on load.
1989: Feature 6: Carry Forward with Reason
1990: Architecture
1991: Carry-forward creates new plan_tasks in the target day_plan. The original task in the source day_plan remains unchanged (its is_done stays false). The new plan_task has: carried_from set to the source date, carry_reason set to the user-provided text (or null).
1992: If the target day_plan does not exist for the carry-to date, it is created automatically. The carried task retains its original title, category, estimate, friction, and slot. If the section already has tasks, the carried task is appended at the end.
1993: Carry reason visibility: shown in the source day's review as "↑ from yesterday — [reason]" in --text-ghost below the title. The reason is also visible in the target day's expanded task view as "↑ Carried from [date] — [reason]".
1994: Soft Intelligence integration: same task carried 3+ consecutive days triggers a suggestion: "You have carried '[title]' 3 days — want to break it into smaller pieces?" (see Section 27).
1995: Bulk carry: the "Carry All" option in the carry sheet carries every unfinished task. If any of those tasks were already carried multiple times, the Soft Intelligence check runs after bulk carry completes.
1996: Morning nudge banner if yesterday had unfinished tasks. User selects which tasks to carry. Optional one-line reason per task ("Waiting on feedback", "Rescheduled"). Reason visible in review as a tiny ghost note. Helps identify patterns over time without journaling.
1997: Feature 7: Recurring Tasks

2074: Feature 19: Global Undo System
2075: Core Philosophy (Updated)
2076: The Undo system is not just a feature — it is a fundamental philosophical choice.
2077: Most apps protect users through warnings, confirmations, and modal interruptions. Flow protects users through reversibility.
2078: Instead of constantly asking "Are you sure?", Flow assumes humans make small mistakes, corrections should feel effortless, and interaction flow should remain uninterrupted.
2079: The safest interface is not the one that asks permission constantly. It is the one that makes recovery effortless.
2080: Architecture
2081: The Undo system maintains an in-memory action stack of the last 50 actions per session. Each action record stores: action type, affected entities (task IDs, plan IDs), previous state (full JSON snapshot), new state (full JSON snapshot), timestamp.
2082: Undo flow: user performs action → action is committed (database write + optimistic UI) → action record is pushed to the stack → Undo snackbar appears at the bottom of the screen → if the user taps Undo within 6 seconds, the action is reversed (previous state restored). If the snackbar auto-dismisses, the action record remains in the stack but is marked as "extended" — the user has 30 seconds to tap a persistent Undo button in the bottom-right corner.
2083: Actions that support Undo: task creation (undo deletes the task), task deletion (undo restores the task with full metadata and position), task completion (undo marks it incomplete), task uncompletion (undo marks it complete — the original state is restored), task archive, task duplicate, task move, task reorder (undo restores the previous order for all affected tasks), carry-forward (undo removes the carried task from the target day), category deletion (undo restores the category and re-assigns it to tasks that had it), template deletion (undo restores the template), Flow Section change (undo moves the task back to the previous slot and position).
2084: Actions that do NOT support Undo because they are not destructive: task title edit (no undo — the edit is the new state), theme change, Flow Section rename, day_start_hour change, notification settings change.
2085: Multiple undo: successive Undo taps reverse multiple actions in reverse chronological order. Each Undo shows an undo snackbar: "Undid [action]." with [Undo again] and [Dismiss].
2086: Stack overflow: when the 50-action limit is reached, the oldest action record is dropped from the stack. The snackbar for the dropped action is also dismissed (the user can no longer undo it). The user is not notified.
2087: Restoration rules: task deletion restoration recreates the plan_task with the original UUID (or a new UUID if the original is in use — the new UUID is transparent to the user). Category deletion restoration re-assigns the category UUID to all tasks that had it. Reorder restoration replays the batch order update with the previous values. The restoration animation is the reverse of the original creation animation (task fades in for deletion undo, card moves back for reorder undo).
2088: Undo snackbar design: fixed at the bottom of the screen, above the bottom nav, below the composer. Background: --bg-overlay with blur. Text: DM Sans 13px, --text-primary. [Undo] button: DM Sans 13px, --accent, bold. No icon. Auto-dismisses after 6 seconds with a slow opacity fade (1s). Multiple snackbars from rapid actions are queued — only the most recent is shown, with "+N more actions" appended if there are queued ones.
2089: Purpose: Prevent the app from ever feeling dangerous or punishing. The Undo system appears after every destructive or state-changing action.
2090: Actions that support Undo:
2091: Task deletion
2092: Task completion / uncompletion
2093: Recurring task removal
2094: Carry-forward dismiss
2095: Category deletion
2096: Template deletion
2097: Bulk carry actions
2098: Interaction Design:
2099: A small snackbar/toast appears at the bottom of the screen
2100: Example: "Task deleted" [Undo]
2101: Snackbar auto-dismisses after 4–6 seconds
2102: Multiple undo actions should queue gracefully
2103: Undo restores task position, slot, metadata, and animations
2104: Motion & Feel:
2105: Snackbar slides up softly from the bottom
2106: Never interrupts the user

2108: Undo dramatically increases trust and reduces anxiety. It supports the philosophy of Flow being forgiving rather than strict.

2150: Feature 22: Advanced Day Reset Logic
2151: Architecture
2152: The day reset logic is one of the most architecturally critical systems in Flow. It determines which "day" any given timestamp belongs to, and this affects everything: review aggregation, carry-forward, recurring task generation, notification scheduling, weekly view grouping.
2153: Core rule: the logical day boundary is defined by day_start_hour (D), not midnight. A timestamp T belongs to day D' if T >= D'[D_start] and T < (D'+1)[D_start]. For example, if D=6: May 22 at 2:30am belongs to May 21. May 22 at 6:00am belongs to May 22. May 22 at 11:59pm belongs to May 22. May 23 at 5:59am belongs to May 22.
2154: Implementation in Supabase: plan_tasks and task_instances store date at the day_plan level (not timestamp). The date field represents the "logical day" per the user's day_start_hour at the time the task was created. When day_start_hour changes, existing tasks' logical day is NOT recalculated (they stay in their original logical day). Only new tasks use the new boundary.
2155: Focus session crossing midnight: the focus session's elapsed time is continuous across the boundary. The session belongs to the logical day in which it started. If the session starts at 11:30pm on May 22 (D=6), it belongs to May 22 even if it crosses to May 23. Completion is attributed to May 22.
2156: Review aggregation: the review for a given date includes all tasks whose day_plan.date matches that date. This uses the logical day, not the calendar day. If the user changes day_start_hour and creates new tasks, those tasks go into the new logical day structure.
2157: Weekly view grouping: same as review — uses logical day. The week shown in the weekly view is the logical week (7 consecutive logical days starting from the most recent Sunday or whatever day the user considers week start).
2158: Purpose: Ensure the app behaves naturally around midnight and unconventional schedules.
2159: Required Edge Cases:
2160: Focus session crossing midnight
2161: Task completion after midnight but before user-defined day start
2162: Carry-forward timing
2163: Notification scheduling
2164: Recurring task generation
2165: Core Rule: The user-defined day start time determines the logical boundary of the day.
2166: Example: Day start = 6am. A task completed at 2:30am still belongs to the previous day.
2167: Incorrect day handling breaks emotional trust in productivity software.

2319: 51. EDGE CASES
2320: Day Boundary & Midnight Handling
2321: Midnight crossing during Focus Mode: if a focus session starts at 11:30pm and crosses midnight, the session belongs to the start date. Elapsed time continues uninterrupted. Task completion after midnight but before day_start_hour: completion is attributed to the previous day. Task completion after day_start_hour: completion is attributed to the current day. Carry-forward banner checks day_start_hour boundary, not calendar midnight.
2322: Recurring Task Edge Cases
2323: If a recurring task is created on a day that already has an instance: no duplicate is created. If day_start_hour changes: existing recurrence schedules are not retroactively modified — they apply from the change forward. If a recurring task is carried forward: the carried instance is a one-off plan_task, not a new recurring instance. Deleting the base recurring task: all future instances are deleted (with Undo support). Past instances remain in review (read-only). Instance marked done before recurring rule change: the instance status is preserved.
2324: Carry-Forward Edge Cases
2325: Task carried 3+ consecutive days: Soft Intelligence suggests breaking it into smaller pieces. Task carried but completed on the target day: the carry reason is preserved in review but the completion overrides the carried status. Dismissed carry-forward banner at noon: the unfinished tasks remain on yesterday's review, with no automatic carry. Bulk carry with some already completed: only truly unfinished tasks appear in the carry sheet. Day start changes while tasks are in carry-pending state: carry banner reapplies at the new day_start_hour.
2326: Sync & Offline Edge Cases
2327: Offline task creation while same task is being created on another device: last-write-wins by timestamp (within 500ms tolerance — both accepted if timestamps differ by < 500ms). Sync queue exceeds 100 items: oldest items are compressed (batched actions on the same task_id are merged). IndexedDB full: show subtle persistent warning, stop queueing, prompt user to export/clear. Supabase connection restored after 24h+ offline: full re-sync from server (local queue wins for items in queue, server state wins for everything else). Conflict between local edit and server edit for same task at same second: server wins, local change moved to top of sync queue for re-application.
2328: Focus Mode Edge Cases
2329: Phone call interrupts focus session: timer pauses automatically via Page Visibility API. Resume prompt appears when user returns. Multiple tasks in focus mode queue: user marks one done, prompt offers "Next task" which loads the next highest-priority unfinished task from the current section. Timer fires while app is backgrounded: notification fires as a local push alert. If notification permissions denied: no alert, elapsed timer resumes on return. Breaking focus without completing: tap back arrow or swipe down. A soft "Task still unfinished" note appears for 2s, then returns to plan. No guilt.

Session expires (token refresh fails): user stays in offline mode, "Sign in again" prompt on any sync operation.

2419: 54. FLOW SECTIONS — CUSTOM RHYTHM SYSTEM
2420: Core Philosophy
2421: Flow replaces fixed Flow Sections (Morning / Afternoon / Evening / Night) with Flow Sections — customizable, named, emotional containers for your daily rhythm. This is not a scheduling grid. It is a way to organize time by how life feels, not by what the clock says.
2422: Traditional planners ask: "What time is it?" Flow asks: "What part of your day is this?"
2423: A rigid clock-based structure quietly contradicts Flow's philosophy of softness, adaptability, emotional rhythm, and humane planning. Flow Sections make the app feel deeply personal, emotionally intelligent, non-corporate, and identity-driven.
2424: What Is a Flow Section?
2425: A Flow Section is a named period of your day with its own emotional identity. Each section has:
2426: Name: Freeform text — "Quiet Start", "Focus Window", "Creative Drift", "Low Energy", "Reset Period", "Night Wind-Down"
2427: Time Range: Flexible start and end times. Supports cross-midnight ranges. Drag-adjustable in the UI.
2428: Atmosphere Color: Soft tint only — muted amber, faded indigo, soft green, warm gray. Never saturated.
2429: Optional Icon: Very restrained set — moon, leaf, spark, coffee, wave, candle, star, cloud, droplet, feather. No emoji-heavy feeling.
2430: Energy Type (optional): Deep Focus, Light Tasks, Recovery, Creative, Social, Reflection. Not visible aggressively — helps Flow understand rhythm for future smart behavior.
2431: Order: The sequence defines vertical order in the timeline.
2432: Why This Fits Flow Perfectly
2433: This system supports: emotional planning, ADHD-friendly structure, flexible lifestyles, night schedules, freelancer routines, student routines, and burnout recovery — without forcing corporate time blocks, "productive hours", or rigid routines.
2434: The old structure assumes: normal sleep schedule, standard work culture, daytime productivity. That alienates students, creatives, night owls, ADHD users, shift workers, and freelancers. Flow Sections are inclusive by design.
2435: Examples of Flow Sections
2436: Focus Time — Deep Work — Quiet Start — Slow Morning — Recovery — Creative Window — Admin Energy — Calm Hours — Night Drift — Reset Period — High Momentum — Wind Down — Brain Fog — Overflow — Social Time — Reflection — Learning Block — Free Time
2437: Every user's set of sections is unique. There is no default template beyond a gentle starting suggestion.

2535: 55. AMBIENT MODE — ADAPTIVE ATMOSPHERE SYSTEM
2536: Core Philosophy
2537: Most productivity apps are visually static. Whether the user is overwhelmed, calm, focused, exhausted, or reflective, the interface always behaves exactly the same. Flow's Ambient Mode changes that. Instead of a fixed interface, Flow becomes an emotionally responsive environment.

2539: The goal: the app should feel emotionally appropriate for the current moment. The user should rarely consciously notice the system changing. They should simply feel calmer, clearer, more grounded, and less pressured.
2540: Ambient Mode is not a gimmick, a visual effect pack, or a dynamic wallpaper system. It is environmental comfort design.
2541: Core Design Principle
2542: The interface should behave more like lighting in a room, weather, atmosphere, or physical space than software dashboards. Changes should feel slow, natural, and almost subconscious.

2944: 59. MOBILE GESTURE FAILURE STATES
2945: Core Philosophy
2946: Fluid, Forgiving Interaction
2947: Gesture-heavy interfaces become frustrating without recovery systems. Flow gestures must feel stable, forgiving, and low-pressure.
2948: Required Recovery Behaviors
2949: Drag Cancellation: if user drags away, the task softly returns to its original position.
2950: Edge Scrolling: timeline scrolls naturally during long drags.
2951: Gesture Thresholds: prevent accidental swipes with minimum movement requirements.
2952: Hold Delays: prevent unintended destructive actions by requiring longer holds.
2953: Emotional Goal: interactions should feel physically trustworthy.
2954: Gesture-driven apps feel magical when they work and frustrating when they don't. Flow's gesture system must include explicit failure and recovery states. Every gesture must have a clear cancel path, a visual hint of what will happen, and a graceful fallback.
2955: Accidental Swipe Handling
2956: Swipe detection requires minimum 40px horizontal movement before any action is committed. Under 40px: the card bounces back to position with a spring animation (200ms). No action taken. If the user's finger moves vertically more than 15px during the swipe gesture: gesture is cancelled and treated as scroll. The user sees the card snap back. Visual feedback during the first 40px: the card subtly shifts (max 8px horizontal offset) and the background action hints appear at 20% opacity. This gives the user a preview of what the gesture does without committing anything.
2957: Drag Cancel Thresholds
2958: Long press initiates drag mode after 300ms. If the user lifts their finger before 300ms: no drag, treated as a tap. During drag: if the user moves less than 10px total in 500ms after lift, the card cancels the drag and returns to its original position. This prevents accidental drags from taps with slight movement. The drag ghost (placeholder outline) remains for 800ms after a canceled drag before fading out. This gives the user a moment to understand what almost happened.
2959: Gesture Conflicts
2960: Primary gesture detection order: 1) Vertical scroll, 2) Horizontal swipe, 3) Long press, 4) Tap. Vertical scroll always wins over horizontal if the user's finger moves vertically within the first 100ms of touch. This prevents the task list from fighting scroll vs swipe. If horizontal and vertical movement are both detected simultaneously (diagonal swipe): the gesture is cancelled and neither scroll nor swipe fires. Edge-scroll during drag: if the user drags a card to the top or bottom edge of the screen, the list auto-scrolls at a rate proportional to the distance from the edge. Auto-scroll rate: 1 card per second at minimum distance, 3 cards per second at maximum edge proximity. A 40px activation zone at top and bottom of the viewport.
2961: Long-List Drag Behavior
2962: When dragging a card in a list longer than the viewport, the list auto-scrolls as the card approaches the top or bottom edge. The dragged card stays under the user's finger (it does not scroll with the list). A visual indicator (scroll position thumb) appears on the right edge during auto-scroll. Auto-scroll continues as long as the card is held near the edge. Release cancels auto-scroll immediately. If the drag is released during auto-scroll, the card drops at the current scroll position. Performance: auto-scroll must not drop below 30fps. On low-end devices, reduce auto-scroll rate by 50% and disable edge-glow effects.
2963: Mobile Browser Touch Quirks
2964: iOS Safari: 300ms tap delay is eliminated via touch-action: manipulation CSS rule. Passive touch event listeners on scroll containers to prevent Chrome's scroll-to-top interference. Android Chrome: overscroll-behavior: contain on the task list to prevent pull-to-refresh from conflicting with drag. Touch callout (long-press menu) disabled via -webkit-touch-callout: none. User-select: none on task cards to prevent text selection during drag. All gesture handlers use pointer events (pointerdown, pointermove, pointerup) — not touch events — for unified mouse+touch+pen support.
2965: Gesture Accessibility Fallbacks
2966: Every gesture action has a visible tappable alternative. Swipe actions: available via tap → expanded view → action buttons. Drag reorder: available via long press → menu → "Move up" / "Move down" buttons. Keyboard alternatives: Tab navigation between cards, Space to toggle expand, Enter to complete, Ctrl+Arrow to reorder. Screen reader: all drag actions announced as "Moveable. Use actions menu to reorder." Gesture hints shown once on first use: a subtle animation showing the gesture (reduced motion-respecting). User can dismiss hints permanently.
2967: Testing Gesture Robustness
2968: Each gesture must pass 100 test cycles on real devices (iPhone 12+, Pixel 6+, iPad, desktop Chrome/Firefox/Safari). Automated gesture testing via Playwright touch simulation. Manual testing on low-end Android (Moto G series) to verify frame rate. Edge case: user wearing gloves. Pointer events must still register (capacitive touch may fail — ensure all actions have button alternatives). Edge case: palm rejection. If contact area > 20mm diameter, ignore the touch.

2973: Core Performance Targets
2974: Mobile: 60 FPS target, under 16ms interaction response, instant task creation, smooth drag behavior.
2975: Three-Tier Rendering System (from day one, not an afterthought): Tier 1 — Full (flagship devices, desktop): All ambient layers active, backdrop-filter blur enabled, canvas dandelion with full breathing animation, spring physics on all gestures, particle effects, grain texture. Tier 2 — Standard (mid-range devices, default PWA): Background color transitions only (no layered blur shapes), backdrop-filter disabled (solid backgrounds for composer and sheets), canvas dandelion active but particle effects off, spring physics on gestures but reduced stiffness, no grain texture. Tier 3 — Minimal (low-end devices, reduced motion preference): Static background color (time-based color only, no animation), all backdrop-filter disabled, dandelion icon is static SVG (no canvas animation), no spring physics (linear ease only), no particles, no grain, no stagger animations. Detect tier using navigator.deviceMemory, navigator.hardwareConcurrency, and prefers-reduced-motion media query. Do not rely on benchmark tests — they are too variable. Default to Tier 2 for anything not clearly flagship. Default to Tier 3 when prefers-reduced-motion is set. Apply at app initialization and do not change mid-session.
2976: Low-End Devices: Flow must gracefully reduce blur, atmosphere, motion, particle density, and shadow complexity. Never break.
2977: Adaptive Performance Layer
2978: performance_mode = { reduced_blur, reduced_motion, simplified_layers }
2979: Atmosphere should NEVER compromise readability, responsiveness, or battery life.
2980: Flow's ambient design is performance-intensive. Blur, particle effects, layered backgrounds, and spring animations must never compromise responsiveness. Concrete budgets ensure the app feels smooth on mid-range devices, not just flagships.
2981: Frame Rate Targets
2982: Target: 60fps on mid-range devices (Snapdragon 700 series, Apple A13+). Minimum acceptable: 30fps on low-end devices (Snapdragon 600 series, Apple A11). Below 30fps: disable ambient layers progressively (background → particles → bloom → grain). Frame rate is measured via performance.now() in Chrome and Instrumentation in Safari. Below-target devices are detected on first session via a 2-second animation benchmark, and the appropriate ambient tier is locked for the session.
2983: Animation Concurrency Limit
2984: Maximum simultaneous animated elements: 12 total. Interactive animations (gesture feedback, transitions): max 4 at once. Ambient animations (background breathing, particles): max 8 at once. If a new animation would exceed the limit, the oldest ambient animation is stopped (not the interactive one). Page transitions override all other animations — ambient pauses during transitions.
2985: Blur and Filter Performance
2986: Maximum blur radius: 20px at rest, 12px during animation. Maximum number of elements with backdrop-filter: 2 simultaneously (composer + active bottom sheet). On low-end devices: disable backdrop-blur entirely, fall back to solid --bg-elevated backgrounds. will-change: transform declared on all composited layers. No blur on elements that animate on scroll (blur recalculates on every frame and destroys scroll performance). SVG filters preferred over CSS filters for grain texture (GPU-accelerated in more browsers).
2987: Paint and Layout Budget
2988: No layout-triggering animations. All animations use only opacity and transform. Layout shifts are forbidden during transitions. Use transform: translateZ(0) to promote all animated elements to their own compositor layer. Maximum DOM node count: 500 on the home screen (including ambient layers). Task lists exceeding 50 items: virtualize with IntersectionObserver (only render visible tasks + 5 buffer above/below).
2989: Memory Budget
2990: Peak memory usage: under 150MB on mobile, under 300MB on desktop. IndexedDB storage: under 50MB for sync queue and cache. Image assets (category icons, avatars): lazy-loaded, max 200KB each. No Web Workers for ambient animation (overhead not justified). If memory exceeds budget, ambient layers are progressively disabled (grain first, then bloom, then particles).
2991: Network Budget
2992: Initial page load (HTML + CSS + JS): under 200KB gzipped. Font loading: Fraunces + DM Sans + JetBrains Mono = ~60KB total. Fonts loaded with font-display: swap to prevent invisible text. Supabase query for initial data: under 5KB. Sync queue flush: max 50KB per batch. All API calls are gzipped by default. PWA service worker caches: font files, app shell, current day's plan data.
2993: Low-End Device Detection
2994: Detected via: device memory (navigator.deviceMemory < 4GB), hardware concurrency (navigator.hardwareConcurrency < 4), GPU benchmark (2-second canvas render test). Once detected: disable all ambient layers permanently during the session (re-check on next session). Disable spring physics (replace with 200ms linear ease). Reduce blur radius to 0. Disable particle effects. Disable stagger animations. Maintain full interactive responsiveness — gesture feedback, task creation, and transitions must remain at 60fps even on low-end devices.
2995: 61. ERROR-STATE PHILOSOPHY
2996: Core Philosophy
2997: Failure Without Anxiety
2998: Most apps treat errors like technical faults, warnings, and system failures.
2999: Flow treats errors as moments of reassurance.
3000: The app should never make users feel panic, uncertainty, or fear of data loss.
3001: Even during failure, the app should feel calm, stable, and trustworthy.
3002: Tone Rules
3003: Never say: "Error occurred" or "Sync failed."
3004: Instead: "Some changes are still catching up. Your work is safe."
3005: Error Categories
3006: 1. Sync Delay — soft syncing indicator, passive retry, no interruption.
3007: 2. Offline State — Flow transitions gracefully into quiet offline mode: "Offline for now. Changes will sync when connection returns."
3008: 3. Conflict Resolution — never expose technical merge jargon. Instead: "Two versions of this task were updated. Choose the version you'd like to keep."
3009: Visual Philosophy
3010: Error states should reduce stimulation, avoid aggressive red, and remain emotionally calm. Prefer muted amber, soft neutral tones, and restrained motion.
3011: The app must NEVER imply "Your work may be lost" unless absolutely true. Trust is everything.
3012: Flow's emotional identity is built on safety, calm, and trust. Error states must never break that trust. Every error message, every failure state, every unexpected behavior must feel like the app is on the user's side — not blaming them, not panicking, not giving up.
3013: Error Language Principles
3014: Never use: "Error", "Failed", "Critical", "Alert", "Warning", "Problem", "Issue". Use instead: "Something didn't go through", "Couldn't save this one", "That didn't work ", "Let's try that again". Every error message must include a clear action the user can take. Every error message should end with a period — no exclamation marks. Never show raw error codes, stack traces, or technical details unless the user explicitly requests them via a "Details" link.
3015: Sync Conflict Message
3016: Scenario: same task edited on two devices while offline. Detection: last-write-wins by timestamp. The losing edit is not discarded — it is saved in a conflict log. UI message: "Another change was saved first. Yours is saved separately." User can review their version vs the server version by tapping "Review changes" in Settings → Advanced. The conflict log shows: timestamp, device, and content of both versions. User can choose to re-apply their version (creates a new task with "(your copy)" appended).
3017: Export Failure
3018: Scenario: export fails due to data size, IndexedDB error, or browser limitation. Message: "Couldn't generate the export right now. You can try again, or your data is always safe with us." Actions: [Try Again] [Download Partial Export] (exports what is available). Partial export includes a note in the file header listing any omitted sections. If export fails repeatedly (>3 attempts), suggest the user contact support via a discreet link.
3019: Authentication Expiration
3020: Scenario: session token expires while user is actively using the app. Detection: API call returns 401. Behavior: app does NOT log the user out immediately. A soft banner appears at the top: "Your session expired. Tap to sign in again." User can continue browsing their currently loaded data (including the current day's plan). Any create/update/delete actions are queued to IndexedDB with a "pending sign-in" flag. When user signs back in, the queue is flushed. If the user ignores the banner for 30 minutes: a second more prominent banner appears. After 60 minutes: app clears the session and returns to the landing screen. The queue is NOT discarded — it is preserved for the next sign-in.
3021: IndexedDB Corruption Recovery
3022: Detection: app stores a schema version hash in IndexedDB on every write. On app start, the hash is validated. If the hash doesn't match or IndexedDB fails to open: behavior depends on online status. If online: silently clear IndexedDB and re-fetch all data from Supabase. Show no message to the user (the app just loads a bit slower than usual). If offline: show a calm message: "Local storage needs a refresh. Please connect to the internet to restore your data." The app shows a simplified UI with only the composer (tasks are created but cannot be saved until online). Once online, IndexedDB is rebuilt and queued actions are flushed.
3023: Stale Session / Server Downtime
3024: Scenario: Supabase is down or unreachable. Detection: health check endpoint fails 3 times in 30 seconds. Behavior: app switches to offline mode automatically. Banner: "We're having a moment. Your data is safe and will sync when we're back." All actions queue locally. The ambient background continues normally — the app never appears broken. When server recovers: sync queue flushes, banner disappears. If downtime exceeds 24 hours: an additional message appears: "We're still working on it. Your local data is preserved." No data is ever lost during server downtime.
3025: Unauthorized Access / RLS Violation
3026: Scenario: user attempts to access another user's data via URL manipulation. Backend: RLS policies reject the query. Frontend: silently redirect to home screen. No error message. No alert. The app behaves as if the data simply doesn't exist (which, from this user's perspective, it shouldn't).
3027: Generic Catch-All Error Handling
3028: Any unhandled error that reaches the UI boundary: caught by a React Error Boundary. The boundary renders a minimal fallback: centered Fraunces message "Something unexpected happened." with a [Reload] button. The ambient background is still visible behind the fallback — the app does not go to a white screen of death. The error is logged to console (dev) or a remote logging service (prod). User contact: no automatic prompt. If the error occurs more than 3 times in a session: show [Contact Support] link with an email draft pre-filled with app state info.
3029: 62. NOTIFICATION TIMING RULES
3030: Core Philosophy
3031: Gentle Continuity, Not Interruption
3032: Notifications should feel like soft awareness, not productivity surveillance.
3033: Notification Types
3034: Soft Reminder: gentle awareness. Example: "Your evening reflection is waiting quietly."
3035: Important Reminder: reserved for intentional deadlines and user-marked priorities.
3036: Quiet Hours: Flow automatically reduces sound, vibration, and visual urgency during night sections, Focus Mode, and calm periods.
3037: Missed Reminder Behavior: Never "You missed 6 tasks." Instead: "Today shifted a little. You can continue whenever you're ready."
3038: Notifications in Flow are gentle reminders, not demands. They should feel like a quiet tap on the shoulder, not an alarm clock. The notification system is designed to maximize usefulness while minimizing interruption.
3039: Batching Rules
3040: Maximum 1 background notification per day total (system-level). In-app notifications (not system notifications) are not subject to this limit — but should still be used sparingly. If multiple notification triggers occur at the same time (e.g., daily nudge + focus window alert): show only the most contextually relevant one. Priority order: focus window alert > carry-forward reminder > daily nudge > plan reminder > soft continuity message. Batched notifications are delivered as a single notification: "3 things to check: Focus time for 'Review PRs' ended. 2 tasks from yesterday waiting. Tap to see."
3041: Quiet Hours
3042: Default quiet hours: 10pm to 7am (user-configurable in Settings). During quiet hours: no system notifications are fired. In-app notifications are suppressed (no banners, no sounds, no haptics). The Drift button badge still updates (visual-only, no alert). Quiet hours respect the user's day_start_hour setting — quiet hours end automatically at day_start_hour. If a quiet-hours boundary is crossed during a focus session: the timer-end notification is suppressed until quiet hours end. Timezone changes: quiet hours adjust to the new timezone immediately. If the user changes timezone to a location where quiet hours would be active at the current local time, the currently scheduled notifications are postponed.
3043: Timezone Travel
3044: When device timezone changes by more than 1 hour: all scheduled notifications are recalculated. Daily nudge adjusts to the new local morning. Carry-forward banner adjusts to the new local noon. Focus window alerts adjust to the new local time (a 3pm window becomes a 3pm in the new timezone). If the timezone change results in a "day jump" (e.g., crossing the International Date Line): the skipped day is marked as "No plan" in the weekly view, and no carry-forward is triggered. The day Pulse acknowledges: "Welcome to a new timezone. Let's start fresh." Notification schedule is rebuilt from scratch after a timezone change of 3+ hours.
3045: Recurrence Reminder Logic
3046: Recurring tasks do NOT generate automatic notifications. Notifications for recurring tasks are opt-in per task (during creation or editing). If enabled: the user chooses a time for the reminder (default: 30 minutes before the task's focus window start, or 9am if no focus window set). The reminder fires at the chosen time on each qualifying day. If the recurring task is marked done before the reminder fires: the reminder is cancelled for that day. If the user carries a recurring task forward, the carry takes precedence and no additional reminder fires.
3047: Adaptive Notification Suppression
3048: If the user consistently dismisses the daily nudge without acting (>5 consecutive days): the nudge is automatically suppressed for 7 days. Exception: if the user has 3+ unfinished tasks consistently, the nudge remains active. If the user completes all tasks before the daily nudge fires: the nudge is cancelled for that day. If the user is actively using the app (any interaction in the last 30 minutes): the daily nudge is suppressed (no need to remind someone already in the app). Focus window alerts still fire even if the user is active — they mark the end of a planned window.
3049: Missed Notification Handling
3050: If a notification was scheduled but the app was closed or the device was off when it fired: the notification is NOT re-fired. Instead, a soft continuity message appears on next app open: "You had a notification earlier about 'Review PRs'." The message is shown once and then dismissed. If 3+ notifications were missed cumulatively, only the most recent one is mentioned. Missed focus window alerts are shown as a single line in the Day Pulse: "Your focus window for 'Review PRs' ended while you were away."
3051: System Notification Permission Handling
3052: On first notification trigger: request system permission with a contextual explanation. Copy: "Flow can send you a gentle daily reminder to keep momentum going. No spam, no urgency — just one soft nudge per day." If permission is denied: all system notifications are disabled. In-app banners are used as a fallback (shown only when the app is open). The Notifications settings page clearly shows whether system permission is granted. No repeated permission requests — if denied once, the setting shows "Notifications disabled in system settings" with a link to the OS settings page.
3053: 63. MULTI-DEVICE BEHAVIORAL RULES
3054: Core Philosophy
3055: One Emotional Space Across Devices
3056: Flow should feel like the same space continuing everywhere — not separate disconnected apps.
3057: Active Focus Session Ownership
3058: If Focus Mode is active on desktop, mobile should reflect: "Focus session active on MacBook"
3059: Cross-Device Continuity
3060: When user leaves desktop and opens mobile: the app softly resumes the same Flow Section, same atmosphere, same active task. This continuity matters emotionally.
3061: Flow is a personal app, not a collaborative tool. Multi-device support exists so the user can move between phone, tablet, and desktop seamlessly — not so two people can edit the same plan. The architecture assumes one active user at a time, on one primary device at any given moment.
3062: Simultaneous Edits (Same Account, Two Devices)
3063: Supabase Realtime propagates changes to all connected clients within 500ms. If the user marks a task done on device A, device B receives the update in real time. If the user is editing a task title on device B while device A deletes the same task: device B's next interaction with that task triggers a "This task no longer exists" inline message. The task card fades out of device B's UI (no undo for deletion received from another device). If both devices create different tasks at the same timestamp: both tasks are accepted (different UUIDs). No conflict. If both devices edit the same field of the same task within 2 seconds: last-write-wins by server timestamp (not device clock).
3064: Active Focus Session Ownership
3065: Focus Mode is a single-session experience. It cannot be active on two devices simultaneously. When the user enters Focus Mode on device A: a focus session lock is created in Supabase (focus_active: true, device_id, task_id, started_at). If the user opens device B during an active focus session: device B shows a banner: "Focusing on 'Task Name' on another device." Device B cannot enter Focus Mode until the session on device A ends or is dismissed. If device A loses connectivity during focus: the session persists for 190 seconds before automatically releasing. If the user force-closes the app on device A without exiting Focus Mode: the session persists for 90 seconds, then releases. On next open of device A, the lock release is detected and a continuity message appears: "Your focus session was closed. Want to resume?" The elapsed timer is preserved from the last sync point.
3066: Mobile-to-Desktop Continuity
3067: When the user opens Flow on a new device: the ui_state is restored from Supabase. The last_active_date, collapsed_slots, mode, and Drift button position are all synced. If the user was in the middle of a task creation flow on mobile and opens desktop: the partially-typed task is NOT synced (it was never committed). This is intentional — unfinished input is considered local-only. The user picks up where they left off: same date, same mode, same view. The carry-forward banner shows on the first device that opens the app after day_start_hour, then is dismissed on all devices. The Day Pulse message is generated per-device (same data, but the Pulse text is computed client-side).
3068: Stale Tab Handling
3069: If a browser tab is left open overnight (or for multiple hours): the tab's Supabase Realtime subscription may disconnect due to inactivity. On reconnection: the tab receives a full state sync (not just incremental changes). The tab's ui_state is refreshed from the server. If the stale tab was in Focus Mode when it was abandoned: the server-side focus lock is released after 90 seconds. When the user returns to the stale tab, a banner shows: "Your session was updated from another device." The stale tab's local state is discarded and replaced with the server state. No merge logic needed — the server state is authoritative.
3070: Sync Reconciliation Animations
3071: When a remote change arrives via Realtime subscription: the affected element animates softly (200ms opacity pulse, 10% scale bounce). This makes remote changes visible without being distracting. Multiple simultaneous remote changes: stagger the pulse animation at 50ms intervals per element. If a task was completed on another device, the checkmark fills in smoothly (same 200ms animation as local completion). If a task was added from another device, it fades into the list at the correct sorted position. If a task was deleted from another device, it fades out over 300ms (no snap disappearance).
3072: Device Management (Future)
3073: Settings → Advanced → Devices shows a list of devices that have recently (7 days) accessed the account. Each device entry shows: device name (from user agent), last active timestamp, browser/app version. User can tap "Sign out of this device" to invalidate its session. This is a future feature, but the architecture should support it from launch (session tracking in Supabase).
3074: Revised: Handoff Mechanism (replaces lock mechanism): When the user opens Focus Mode on device B while device A has an active session, show a banner on device B: 'Task Name is active on another device.' Offer a single action: 'Continue here instead.' Tapping this immediately transfers the session to device B — the elapsed time is preserved, device A's session closes gracefully (stores elapsed time to Supabase before releasing). No waiting, no lock expiry, just a one-tap handoff. If the app cannot confirm connectivity within 90 seconds of losing it during focus, release the lock. Add a 'Keep going on this device' option on device B so users who accidentally opened the second device can dismiss the banner without disrupting the primary session.

3110: PART TWELVE — DESIGN PRINCIPLES & FINAL GOAL
3111: 65. UNIVERSAL UNDO SYSTEM — SAFETY WITHOUT INTERRUPTION
3112: Core Philosophy
3113: Most apps protect users through warnings, confirmations, and modal interruptions.
3114: Flow protects users through reversibility.
3115: Instead of constantly asking "Are you sure?", Flow assumes:
3116: * humans make small mistakes,
3117: * corrections should feel effortless,
3118: * interaction flow should remain uninterrupted.
3119: The app should feel forgiving, calm, and safe.
3120: Core UX Principle
3121: The safest interface is not the one that asks permission constantly. It is the one that makes recovery effortless.
3122: Undo Architecture
3123: Every reversible action enters a temporary action stack:
3124: undo_stack = [task_delete, task_complete, task_move, section_edit]
3125: Each action includes: timestamp, previous state, restoration animation, affected objects, and sync state.
3126: Supported Undo Actions
3127: Task Actions: complete, delete, archive, move, reorder, duplicate
3128: Drift Actions: convert to task, delete entry, merge entries, archive
3129: Flow Section Actions: rename, delete, reorder, atmosphere change
3130: Reflection Actions: dismiss reflection, archive reflection
3131: Undo UI Philosophy
3132: Undo should feel lightweight, immediate, and emotionally soft — never alarming, warning-heavy, or system-like.
3133: Restoration Animation
3134: Undo should visually reverse the action. Example: a deleted task softly returns to its original position, opacity restores gradually, spacing reopens naturally. The interface should feel alive and forgiving.
3135: Confirmation Rules
3136: Confirmation dialogs should be rare — only for truly irreversible actions:
3137: * Permanent delete: "Delete permanently? This cannot be undone."
3138: * Account deletion: "Delete your Flow space? Your reflections, Drift entries, and history will be removed permanently."
3139: * Full reset: "Reset all Flow Sections?"
3140: Important UX Rule
3141: Never interrupt task completion, task movement, lightweight edits, or quick interactions. Flow should preserve interaction momentum.
3142: Emotional Goal
3143: The user should feel: "Mistakes are safe here." That emotional trust becomes foundational.
3144: Sync Compensation: Every undo action must trigger a compensating write to Supabase, not just a local state restoration. Task deletion undo — creates the task in Supabase (with original UUID if available). Task completion undo — updates is_done: false in Supabase. Task creation undo — deletes the task from Supabase. Reorder undo — sends a batch order update to Supabase. The undo action enters the same sync queue as forward actions. UI restores instantly (local-first), Supabase compensation fires in background. If the compensation fails, undo is marked as 'pending sync' — the UI shows the restored state with a subtle sync indicator. This means the undo system is not just an in-memory stack — it is a reversible action journal. Each action stores not just the previous state but also a reversal operation that can be written to Supabase. The undo system and the sync queue share a single source of truth about pending operations — implement them as one system (a write journal) rather than two separate systems.

3163: 67. COMPONENT EMOTIONAL STATE SYSTEM — CONSISTENT EMOTIONAL INTERACTION
3164: Core Philosophy
3165: Every component should communicate emotion consistently. Buttons, cards, tasks, toggles, and Drift entries all need emotional interaction rules.
3166: Emotional State Map
3167: Hover: invitation — the component softly acknowledges presence.
3168: Active: grounded — the component feels pressed, stable, certain.
3169: Focused: clear — the component is ready for input, calmly.
3170: Disabled: quiet — the component steps back visually.
3171: Completed: resolved — the component settles with finality.
3172: Archived: distant — the component recedes into the background.
3173: Syncing: transitional — the component breathes gently.
3174: Important Rule
3175: No interaction should feel harsh, abrupt, aggressive, or hyper-productive. Everything should feel soft, breathable, and emotionally coherent.
